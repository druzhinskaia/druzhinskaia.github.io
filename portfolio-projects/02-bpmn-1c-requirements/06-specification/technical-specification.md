# Техническое задание на доработку 1С

## 1. Цель доработки

Автоматизировать процесс обработки заявок в 1С: создать единую форму заявки, добавить обязательные поля, статусы, контроль сроков, уведомления и отчет для руководителя.

## 2. Объект доработки

Документ 1С: `Заявка`.

## 3. Роли пользователей

| Роль | Описание |
|---|---|
| Инициатор | Передает данные по заявке менеджеру |
| Менеджер | Регистрирует, проверяет и ведет заявку в 1С |
| Руководитель | Согласует нестандартные заявки и контролирует просрочки |
| Исполнитель | Выполняет заявку |
| Администратор | Настраивает справочники и права |

## 4. Поля документа `Заявка`

| Поле | Тип | Обязательное | Описание |
|---|---|---|---|
| Номер заявки | Автонумерация | Да | Уникальный номер заявки |
| Дата создания | Дата/время | Да | Заполняется автоматически |
| Инициатор | Справочник | Да | Клиент, подразделение или сотрудник, от которого поступил запрос |
| Тема заявки | Строка | Да | Краткое описание |
| Описание | Текст | Да | Детали заявки |
| Категория | Справочник | Да | Тип обращения |
| Приоритет | Список | Да | Низкий, Средний, Высокий |
| Ответственный | Пользователь | Да | Назначенный менеджер |
| Статус | Список | Да | Текущий статус |
| Контрольный срок обработки | Дата/время | Да | Рассчитывается по правилам SLA |
| Дата закрытия | Дата/время | Нет | Заполняется при завершении |

## 5. Статусы заявки

| Статус | Описание |
|---|---|
| Новая | Заявка создана и ожидает проверки |
| Требует уточнения | Не хватает данных |
| В работе | Заявка принята менеджером |
| На согласовании | Ожидает решения руководителя |
| Согласована | Разрешена к выполнению |
| Отклонена | Не принята в работу |
| Выполнена | Работы завершены |
| Закрыта | Заявка проверена и закрыта |

## 6. Функциональные требования

### FR-01. Документ `Заявка`

В 1С должен быть создан документ `Заявка` с единой формой ввода, обязательными полями и подсказками для пользователя.

### FR-02. Обязательные поля

Поля инициатор, тема, описание, категория и приоритет должны быть обязательными для заполнения.

### FR-03. Запрет сохранения без обязательных полей

Система не должна позволять сохранить заявку без обязательных полей: инициатор, тема, описание, категория, приоритет.

### FR-04. Справочник статусов

Статус заявки выбирается только из утвержденного списка. Ручной ввод статуса запрещен.

### FR-05. Автоматический расчет срока

Контрольный срок обработки рассчитывается автоматически на основании приоритета:

- Высокий: 1 рабочий день.
- Средний: 3 рабочих дня.
- Низкий: 5 рабочих дней.

### FR-06. Назначение ответственного

При создании заявки система должна назначать ответственного менеджера по правилам распределения или выбранному значению в форме.

### FR-07. Уведомление ответственному

Система отправляет уведомление ответственному при создании заявки.

### FR-08. Уведомление руководителю при просрочке

Система отправляет уведомление руководителю при нарушении контрольного срока обработки.

### FR-09. Согласование

Если приоритет высокий или категория требует решения руководителя, заявка переводится в статус `На согласовании`.

### FR-10. История статусов

Все изменения статусов должны сохраняться в истории заявки с датой, пользователем и новым статусом.

### FR-11. Отчет руководителя

Добавить отчет со следующими показателями:

- количество заявок по статусам;
- количество просроченных заявок;
- заявки по ответственным;
- среднее время обработки;
- список заявок с высоким приоритетом.

### FR-12. Ограничение доступа к справочникам

Изменять справочники и списки статусов может только пользователь с ролью администратора.

## 7. Нефункциональные требования

- Интерфейс должен быть понятен пользователям без дополнительного обучения.
- Все изменения статусов должны сохраняться в истории заявки.
- Отчет должен формироваться не дольше 10 секунд при объеме до 10 000 заявок.
- Доступ к изменению справочников должен быть только у администратора.

## 8. Критерии приемки

- Нельзя сохранить заявку без обязательных полей.
- Статус можно выбрать только из справочника.
- Контрольный срок обработки рассчитывается автоматически.
- Уведомление создается при новой заявке и при нарушении контрольного срока.
- Руководитель видит отчет по статусам, срокам и ответственным.
- История изменения статусов сохраняется в карточке заявки.
