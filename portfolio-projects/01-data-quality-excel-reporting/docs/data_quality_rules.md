# Правила контроля качества данных

## Цель

Проверить выгрузки из 1С перед подготовкой управленческой отчетности и снизить риск попадания ошибочных данных в итоговые показатели.

## Проверяемые таблицы

### Заказы

Файл: `data/raw/orders_raw.xlsx`

Ключевые поля:

- `order_id` - идентификатор заказа.
- `order_date` - дата заказа.
- `client_id` - идентификатор клиента.
- `manager` - ответственный менеджер.
- `quantity` - количество.
- `price` - цена.
- `order_sum` - сумма заказа.
- `status` - статус заказа.

Проверки:

- дубликаты `order_id`;
- пустой `client_id`;
- пустой `manager`;
- клиент отсутствует в справочнике клиентов;
- `order_sum` не равна `quantity * price`;
- отрицательная цена;
- дата заказа позже 2026-06-25;
- статус не входит в список: `Выполнен`, `В работе`, `Отменен`, `Ожидает оплаты`.

### Клиенты

Файл: `data/raw/clients_raw.xlsx`

Ключевые поля:

- `client_id` - идентификатор клиента.
- `client_name` - название клиента.
- `inn` - ИНН.
- `city` - город.
- `registration_date` - дата регистрации.

Проверки:

- дубликаты `client_id`;
- пустой ИНН;
- ИНН не равен 10 символам;
- пустой город;
- пустая дата регистрации.

### Оплаты

Файл: `data/raw/payments_raw.xlsx`

Ключевые поля:

- `payment_id` - идентификатор оплаты.
- `order_id` - идентификатор заказа.
- `payment_date` - дата оплаты.
- `payment_sum` - сумма оплаты.
- `payment_status` - статус оплаты.

Проверки:

- пустой `order_id`;
- заказ отсутствует в таблице заказов;
- сумма оплаты больше суммы заказа;
- пустой статус оплаты;
- дата оплаты раньше даты заказа;
- статус не входит в список: `Оплачено`, `Частично оплачено`, `Не оплачено`.

## Логика отчета

1. Исходные выгрузки размещены на листах `01_orders_raw`, `02_clients_raw`, `03_payments_raw`.
2. На листе `04_checks` каждая запись проходит набор формульных проверок.
3. В колонке `error_count` считается количество сработавших проверок по строке.
4. Лист `05_errors` показывает компактный реестр событий ошибок с исходной строкой, типом, критичностью и рекомендацией.
5. Лист `06_summary` агрегирует показатели для руководителя.
6. Лист `07_dashboard` показывает KPI, динамику заказов, ошибки по типам и статусы заказов.
7. Лист `08_action_plan` содержит план действий по исправлению найденных проблем.

Подробная карта расчетов приведена в `docs/formula_catalog.md`.

## Важное уточнение по метрикам

- `04_checks.error_count` показывает количество сработавших проверок в каждой строке.
- `05_errors` показывает бизнес-реестр событий ошибок. Он используется для KPI `Количество ошибок`, разбивки по типам, критичности и объектам.
- `Доля записей с ошибками` считается по строкам, где `error_count > 0`, поэтому это отдельная метрика качества записей.

## Ограничения

Данные синтетические. Проект не использует реальные персональные, клиентские или финансовые данные.
