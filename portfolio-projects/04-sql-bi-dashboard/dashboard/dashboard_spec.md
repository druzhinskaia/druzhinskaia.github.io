# Спецификация дашборда

## Целевая аудитория

Руководитель отдела продаж и коммерческий директор.

## Управленческий сценарий

Дашборд используется еженедельно для контроля продаж, маржинальности и дебиторской задолженности. Руководитель должен за 3-5 минут понять, выполняется ли план по выручке, какие категории дают вклад, где проседает маржинальность и какие клиенты требуют внимания по оплатам.

## Блоки дашборда

- KPI-карточки: выручка, заказы, средний чек, маржинальность, дебиторская задолженность, топ-категория.
- Динамика выручки по месяцам.
- Выручка по категориям.
- Клиенты с максимальной задолженностью.
- Канал продаж с максимальной выручкой.
- Интерактивный фильтр по каналу продаж в автономном HTML-прототипе.

## Метрики и источники

| Блок | Метрика | Источник |
|---|---|---|
| KPI | выручка, заказы, средний чек, маржинальность, дебиторская задолженность | `result/kpi_summary.csv` |
| Динамика | заказы, выручка, прибыль, маржинальность по месяцам | `result/monthly_sales.csv` |
| Категории | заказы, выручка, прибыль, маржинальность | `result/category_performance.csv` |
| Задолженность | клиент, сегмент, регион, сумма задолженности, количество неоплаченных заказов | `result/receivables_by_customer.csv` |
| Каналы | заказы, выручка, средняя выручка строки, рейтинг канала | `result/channel_performance.csv` |

## Фильтры для BI-версии

- период заказа;
- канал продаж;
- категория товара;
- регион клиента;
- сегмент клиента;
- статус оплаты.

## Правила расчета

- В расчет продаж включаются только неотмененные заказы.
- Выручка считается по строкам заказа: `SUM(order_items.line_revenue)`.
- Прибыль считается как `line_revenue - line_cost`.
- Маржинальность считается как `SUM(profit) / SUM(revenue)`.
- Дебиторская задолженность считается как сумма `revenue - paid_amount` по заказам, где оплата меньше суммы заказа.
- Рейтинг каналов считается оконной функцией `RANK()` по выручке.

## Решения для визуализации

- KPI вынесены в верхний ряд для быстрого чтения.
- Категории и задолженность показаны горизонтальными барами, потому что названия длинные.
- Динамика выручки показана линией по месяцам.
- Клиенты с задолженностью выделены отдельным блоком, потому что это зона управленческого действия, а не просто аналитическая справка.
- Топ-канал показан как отдельный callout, чтобы быстро отделить канал-лидер от общей динамики.

## Ограничения

- Данные синтетические и отражают типовую структуру интернет-магазина компьютерного оборудования.
- Дашборд представлен как SVG-макет, спецификация и автономный HTML-прототип. HTML-файл демонстрирует BI-логику и сценарий фильтрации, но не заменяет промышленную BI-систему.
- Для продуктовой версии потребуются роли доступа, регламент обновления и согласованный порог критичной задолженности.

## Регламент обновления

- Периодичность: еженедельно.
- Ответственный за подготовку данных: аналитик.
- Пользователь результата: руководитель отдела продаж.
- Контроль качества перед публикацией: сверка количества заказов, суммы выручки и суммы оплат с исходной базой.
